# Hook 강화 + 설계 무관용 원칙 논의 (S31, 2026-03-21)

## 사용자 확정 원칙

### hook 설계 시 구멍 금지
- 임계값을 느슨하게 두면 규칙 우회 구멍이 됨
- 사용자: "왜 10개로 해놓냐는거야 니가 애초에 설계할 때부터 구멍을 만드네"
- 원칙: hook 조건은 최대한 엄격하게, 예외는 명시적으로만 허용

### WI 번호 중복 완전 차단
- 기존: 같은 WI로 10개 이상 커밋 시 경고만 출력
- 변경: 이미 머지된 WI 번호 재사용 즉시 차단 (exit 2)
- 사용자가 직접 WI-160으로 커밋되는 문제를 발견하여 지적

### 빈 팀 우회 방지
- 기존: 팀 활성 여부만 확인 (빈 팀도 통과)
- 변경: Guardian + DocOps 필수 멤버 존재 확인
- "빈 팀으로 우회하지 마세요"

## 사용자 피드백 패턴

| 피드백 | 의미 | 대응 |
|--------|------|------|
| "니가 계속 차단에 걸릴때까지 맘대로하는거보면 claude.md에 내용이 빠진거야?" | 규칙 미준수 = 설계 결함 | CLAUDE.md + rules 점검, hook 보강 |
| "왜 10개로 해놓냐는거야" | 느슨한 임계값 = 설계 구멍 | 임계값 0으로 엄격화 |
| "니가 규칙을 따를수밖에 없게만드는게 지금 최우선과제" | 자율 규칙 준수 불가능 → 기계적 강제 필수 | hook으로 원천 차단 |
| "이번에도 WI가 160으로 되어있는데" | WI 번호 분리 위반 반복 | 중복 차단 hook 강화 |

## 변경 사항 (verify-on-commit.sh)

1. **WI 번호 중복 차단** (1-2 섹션):
   - `WI_PR_COUNT > 10` → `WI_TOTAL > 0` (머지+직접 커밋 합산)
   - echo 경고 → exit 2 차단

2. **빈 팀 필수 멤버 확인** (1-4 섹션 신규):
   - jq로 config.json 파싱
   - Guardian 없으면 차단
   - DocOps 없으면 차단
