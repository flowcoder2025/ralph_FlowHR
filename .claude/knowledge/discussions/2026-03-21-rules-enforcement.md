# Rules 강제주입 한계 + 코워크 항상 활성화 논의 (S34, 2026-03-21)

## 사용자 확정 원칙

### rules 강제주입의 한계
- `.claude/rules/` 파일은 세션 시작 시 강제 주입됨
- 그러나 유저 메시지로 취급되므로, 작업 압박이 있으면 무시될 수 있음
- 사용자: "결론적으로 룰스가 강제주입되지만 그건 유저 메시지로 취급되니 작업압박이 있으면 무시되는거네"
- 원칙: rules만으로 강제 불가 → 코워크+hook으로 기계적 강제 보완 필수

### 코워크 항상 활성화 필요
- 사용자: "그럼 코워크가 항상 강제되어야하는거네"
- 사용자: "그럼 항상활성화로 변경시키는게 정답이네"
- 코워크 없이 단독으로 작업하면 rules가 무시될 위험
- 첫 커밋에서 차단하는 방식으로 강제

### 작업 전 파일 숙지 검증
- 사용자: "그럼 작업전 파일을 다안읽고 시작한걸 검증하는 단계도있어야겠네"
- Guardian에 Step 4 "리드 파일 숙지 크로스체킹" 추가
- knowledge/index.md 기반으로 관련 파일 목록 확인 → 리드의 태스크 설명과 대조

## 방향 전환 이력

1. **세션 시작**: S33 hook 작업 완료 후 main으로 머지, 브랜치 정리
2. **사용자 질문**: "신규세션 대화내용 확인해봐" — 새 세션에서 이어서 진행할 때 맥락 유실 여부 확인
3. **문제 발견**: rules 파일이 주입되지만 실제로는 무시될 수 있음
   - 사용자: "내가 지적해서 교정된거고 내가 지적하지않았으면 방향이 틀어질 가능성은 없었나?"
   - 사용자: "룰스 파일이 무시돼? 그건 강제주입아니었어?"
4. **결론**: 코워크 항상 활성화 + Guardian 파일 숙지 검증 추가
5. **WI-165 구현**: Guardian agent에 Step 4 추가 (PR #209)

## 변경 사항

### WI-165: Guardian 파일 숙지 크로스체킹 (PR #209)
- guardian-agent.md에 Step 4 신규 추가
- 검증 절차:
  1. knowledge/index.md에서 현재 작업 주제에 해당하는 파일 목록 확인
  2. 관련 에이전트 정의, 스크립트, 규칙 직접 읽기
  3. 리드의 태스크 설명/설계와 파일 내용 대조
  4. 불일치 시 "[GUARDIAN] 리드가 {파일}을 숙지하지 않은 것으로 보입니다" 보고
