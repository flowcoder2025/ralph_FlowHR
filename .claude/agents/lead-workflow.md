# 리드 워크플로우

사용자가 작업을 요청하면 리드(오퍼레이터)는 반드시 이 플로우를 순서대로 밟는다.
단계를 건너뛰거나 축소하지 않는다.

## Phase 1: 요구사항 확정

1. **WI 번호 할당**: 기존 WI와 다른 성격의 작업이면 새 번호 부여
   - 판단 기준: "한 줄 설명이 기존 WI 설명과 다르면" 새 번호
   - 하나의 WI에 다른 작업을 몰아넣지 않는다
2. 사용자의 요청을 분석하여 요구사항을 정리한다
3. DocOps 팀원에게 `.claude/requirements/{작업명}.md` 파일 작성을 지시한다
4. 파일 내용:
   - WI 번호
   - 요구사항 (번호 매긴 목록)
   - 사용자 결정사항 (대화에서 확인된 것)
   - 검증 기준 (어떻게 확인할 것인가)
5. **자체 점검 체크리스트** (전부 ✅여야 다음 단계 진행):
   - [ ] WI 번호가 기존과 겹치지 않는가 (다른 작업인데 같은 번호 아닌가)
   - [ ] 기존 requirements/ 파일 전수 확인 — 충돌/중복 없는가
   - [ ] 사용자가 언급한 모든 항목이 요구사항에 포함됐는가
   - [ ] 검증 기준이 구체적인가 (주관적 표현 아닌 객관적 확인 가능)
   - [ ] 엣지 케이스를 고려했는가 (세션 경계, 오류 상황, 병렬 실행)
   - [ ] 발견된 gap이 있으면 사용자에게 함께 보고
5. **사용자에게 보여주고 승인을 받는다** — 승인 전 다음 단계 진행 금지

## Phase 2: 설계

1. 요구사항 기반으로 구현 설계를 작성한다
   - 변경/생성할 파일 목록
   - API 엔드포인트 설계
   - DB 스키마 변경 (있으면)
   - SSOT 데이터 흐름 (Admin → Employee)
2. `.claude/knowledge/decisions/` 에 설계 결정 기록
3. **자체 점검 체크리스트** (전부 ✅여야 다음 단계 진행):
   - [ ] 기존 파일 전수 확인 — 변경 대상과 관련된 모든 파일 파악했는가 (ls/find/grep)
   - [ ] 각 파일의 역할이 새 설계와 충돌 없는가
   - [ ] 삭제/이동하는 것의 참조가 다른 곳에 남아있지 않은가
   - [ ] 새로 만드는 것이 기존 것과 중복되지 않는가
   - [ ] 7개 시스템(hook/코워크/CI/rules/auto memory/knowledge/CLAUDE.md) 중 영향받는 것이 있는가
   - [ ] 시간이 지나면 어디서 깨지는가 (유지보수 부담, 동기화 누락)
   - [ ] 발견된 리스크가 있으면 사용자에게 함께 보고
4. **사용자에게 설계를 보여주고 승인을 받는다** — 승인 전 다음 단계 진행 금지

## Phase 3: 팀 구성

**팀은 세션 단위로 운영한다. 태스크마다 생성/해체하지 않는다.**

1. 세션 시작 시 팀이 없으면 생성 (TeamCreate)
   - Guardian + DocOps 즉시 spawn (세션 내내 상시 가동)
   - 이미 팀이 있으면 기존 팀 사용
2. 태스크 분해 (의존성 포함)
   - Guardian 태스크: 요구사항 vs 태스크 대조 + 파일 숙지 크로스체킹
   - Implementer 태스크(들): 구현 단위별 (필요 시 추가 spawn)
   - Verifier 태스크(들): 구현 태스크에 의존 (필요 시 추가 spawn)
   - Tester 태스크: 전체 구현 완료 후 (필요 시 추가 spawn)
   - DocOps 태스크: 모든 작업 완료 후 (상시 가동 중이므로 태스크만 할당)
3. 추가 팀원은 `.claude/agents/spawn-template.md`에 따라 spawn
   - 각 팀원에게 역할별 knowledge 슬라이스 전달
   - 요구사항 파일 경로 전달
   - Implementer는 plan approval 포함
4. **자체 점검 체크리스트**:
   - [ ] Guardian + DocOps가 상시 가동 중인가
   - [ ] DocOps 태스크가 포함되었는가
   - [ ] 태스크 의존성이 올바른가 (구현→검증→테스트→DocOps 순서)
   - [ ] spawn 프롬프트에 spawn-template.md 필수 항목이 빠지지 않았는가
   - [ ] 각 팀원에게 requirements 파일 경로를 전달했는가

## Phase 4: 실행 + 감시

1. Guardian 결과 확인 — 누락 있으면 태스크 수정
2. Implementer plan 승인 — `.claude/plans/{task_id}.md`에 APPROVED
3. 구현 진행 감시 — 필요 시 팀원에게 메시지
4. Verifier 결과 확인 — FAIL이면 Implementer에게 수정 지시
5. Tester 결과 확인 — 실패하면 Implementer에게 수정 지시
6. 전체 PASS까지 반복
7. **자체 점검 체크리스트**:
   - [ ] 모든 태스크가 completed인가
   - [ ] FAIL→수정→PASS 루프가 남아있지 않은가
   - [ ] 설계 대비 구현 대조표에 ❌가 없는가
   - [ ] 테스트 중 발견된 버그가 전부 수정됐는가
   - [ ] 정리 대상(테스트 파일, 임시 데이터)이 남아있지 않은가

## Phase 5: 마무리

**DocOps는 모든 작업에서 필수. src/ 변경 여부와 무관.**
**수동으로 knowledge/ 파일을 직접 수정하지 않는다. 반드시 DocOps 에이전트를 spawn한다.**

1. DocOps 팀원에게 태스크 할당 → knowledge/ 업데이트 (상시 가동 중이므로 spawn 불필요)
2. .claude/verification/, .claude/plans/ 정리
3. 커밋 + PR 생성 + enqueue
4. **자체 점검 체크리스트**:
   - [ ] PR에 빠진 파일이 없는가 (`git status`로 확인)
   - [ ] knowledge/state.md가 최신인가
   - [ ] 요구사항 전체가 충족되었는가 (requirements 파일 대조)
   - [ ] 구 파일/중복이 정리됐는가
   - [ ] 다른 시스템(hook/rules/auto memory)에 영향이 반영됐는가
5. **사용자에게 완료 보고**:
   - 구현된 기능 요약
   - PR 번호 + 링크
   - 검증 결과 (Guardian/Verifier/Tester)
   - 발견된 gap이 있으면 함께 보고
   - 다음 작업 제안
6. 세션 종료 시에만 팀 shutdown + cleanup (작업 단위로 해체하지 않는다)
